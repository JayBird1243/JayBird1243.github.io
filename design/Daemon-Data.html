SaaS data infrastructure case study

Daemon Data

A Firebase and Google Cloud based system for collecting Reddit activity, sharing scrape effort across users, running AI analysis, and turning the results into configurable notifications. The project started as a personal market-signal experiment and grew into a hosted SaaS architecture designed around rate limits, multi-tenant data reuse, and user-defined automation.

Core Idea

Turn individual scraping requests into shared infrastructure.

Daemon Data is built around a central principle: if two users want overlapping Reddit data, the system should collect it once, store it once, and distribute the value many times. Instead of treating each workflow as an isolated job, workflows feed shared scheduler documents that combine user demand into one scrape plan per subreddit.

That design reduced duplicate Reddit API calls, made Firestore storage more efficient, and created a cleaner separation between shared facts and private user data. Central post documents hold canonical Reddit state, while each user keeps a lightweight record of which posts belong to their workflows and what analysis was produced for them.

Shared data layer

Canonical post documents and daily delta histories are readable to authenticated users, avoiding redundant copies of the same Reddit post.

User ownership layer

User post records track workflow ownership, first-seen statistics, selected prompts, and private AI analysis results.

Automation layer

Schedulers, Pub/Sub messages, and Cloud Tasks coordinate recurring scrapes, cleanup, analysis, and notification evaluation.

Authorization

One account model across Google and Discord.

Users can sign in with Google through Firebase Auth or with Discord through a custom OAuth flow. The Discord path exchanges the OAuth code server-side, fetches the verified Discord profile, resolves the correct Firebase user, and returns a Firebase custom token for the browser to complete sign-in.

The important design choice is account continuity. If a Discord login uses the same email as an existing Google account, Daemon Data links the Discord identity to that existing user instead of creating a duplicate. The reverse path is also supported through a linking flow from the profile area. Discord IDs are enforced as one-to-one mappings, so one Discord account cannot silently attach to multiple users.

  • Existing Discord link: find the user document with that Discord ID and mint a custom token for that Firebase UID.
  • Matching email: connect Discord fields to the existing Firebase account and mark the provider-verified email as verified.
  • New Discord user: create a custom Firebase Auth user with a stable discord:{id} UID and initialize their Firestore defaults.
Workflows

User settings become scheduler demand.

A workflow is the user's reusable data-collection recipe. It can include selected subreddits, selected flairs, AI prompt selections, notification selections, scrape cadence, post age window, run duration, and how many posts to retrieve. Post quantity is normalized to 100-post steps between 100 and 1000 because Reddit listing calls are naturally collected in pages of 100.

Customizable workflow settings include selected subreddits and flairs, runtime type, scrape frequency, finite duration, post age window, max retrieved posts, AI enabled state, selected model, selected prompts, per-flair analysis toggles, notification choices, and comment settings such as whether comments are included, top-level comment count, replies per comment, and depth. Advanced scheduling adds admin-gated controls for bot selection, usage budget, frequency, delta duration, and per-user advanced scheduler state.

When a workflow runs, the backend snapshots the user's relevant settings into shared scheduler documents. For normal recurring runs, each schedulers/{subreddit} document aggregates all users interested in that subreddit. It keeps the fastest requested interval, largest requested post quantity, widest post age window, latest end date, comment maxima, selected prompt IDs, and active notification IDs. If those aggregate settings change, the Cloud Scheduler job is recreated with a fresh embedded payload; when users stop or expire, cleanup tasks remove their workflow entry and recompute the aggregate from remaining subscribers.

Data Collection

Block-based scraping with light and heavy write paths.

The scraper pulls Reddit listings in fixed pages of 100 posts. Each page is evaluated against the workflow's post age window and post quantity limit, then written through three coordinated paths: the central post collection, the user's post collection, and the user's analysis queue.

Heavy writes update the canonical post document with fields like subreddit, title, body, author, permalink, flair, score, comment count, scrape time, and the post's detailed content fields. Delta writes record time-series points for upvotes and comment counts under daily subdocuments, including first and second derivative fields when enough history exists. The delta path is intentionally lighter and batched, because it may run repeatedly to track movement over time.

The user post collection stays intentionally small: it records that a user/workflow owns a post, stores the first-seen metadata needed by the dashboard, and later holds analysis results keyed by prompt. The analysis queue is only populated when the workflow enables AI, the relevant prompt IDs are selected, the flair is eligible for analysis, and the user is not blocked by billing state.

Bot Pool

Transactions protect scarce Reddit capacity.

Reddit's practical limit of roughly 100 posts per minute per bot shaped the scraper architecture. Daemon Data treats each bot as a leased resource in Firestore. Scrape workers randomly sample bot pools, query currently available bots, and claim a candidate inside a Firestore transaction. The transaction reads available_at and only writes the lease if the bot is still available, preventing two concurrent scrapes from racing onto the same bot.

Successful scrapes release the bot by moving available_at back to the present. Rate-limited bots are handled differently: the worker moves available_at into the future and does not immediately release it, letting the bot cool down on its own. Authentication failures and repeated forbidden responses can create longer lockouts, protecting the system from repeatedly hammering a bad credential or private subreddit.

Advanced scraping uses Cloud Tasks to split a user's subreddit list across time and a selected bot budget. If a bot is unavailable, the task is rescheduled around that bot's recovery time with jitter, avoiding thundering-herd retries when many tasks are waiting for the same resource.

AI Analysis

Prompt results are queued, billed, and chained.

Analysis means AI-generated replies to user-defined prompts. Scraping does not call the model directly. It writes queue documents and publishes a kickoff message for the batch. The analysis dispatcher then creates Cloud Tasks only up to each user's concurrency capacity, and each worker chains the next queued item when it finishes. This keeps LLM usage controlled without leaving the queue idle.

Users select their desired model in workflow settings or in manual analysis. The worker detects the provider from the model ID and can call OpenAI, Anthropic, or Gemini using keys stored in Secret Manager. Token usage is converted into credits with a pricing table, deducted transactionally, and logged by day and minute. If a user runs out of credits, analysis is blocked before new tasks are created and again inside the worker before model calls continue.

Automated analysis happens during scraping when workflow AI settings and per-flair gates allow it. Manual analysis can be triggered later for selected posts and selected prompts. In reanalysis mode, the queue resets prompt completion so a new result can overwrite previous erroneous analysis while reusing the same post data.

Prompts

A block builder turns user intent into structured model calls.

Prompts are stored as block-based documents rather than plain text alone. A prompt has a name, description, response type, compiled prompt, and ordered blocks. The response type tells the worker how to parse the model's answer: bounded or unbounded numeric, bounded string, bounded multiple string, or unbounded string. Bounds and options are extracted from the prompt blocks and carried into workflow references.

When an analysis worker runs, it loads the selected prompt documents, inserts the current post's title, body, subreddit, upvotes, and comment count into the compiled prompt, calls the chosen model, parses the response according to the response type, and stores the result in the user's post document. The analysis output includes provider, model, timing, token usage, credits charged, parsed value, and bounds where relevant.

Notifications

A graph evaluator turns analysis and metrics into alerts.

Notifications are built as expression graphs. Users connect input blocks, analysis blocks, primitive values, arithmetic functions, boolean comparisons, and output blocks. The evaluator walks backward from the output node, computes the dependency graph, and decides whether a notification should trigger.

The system supports both single-post triggers and time-horizon triggers. Single-post mode can publish immediately when a post satisfies the condition. Time-horizon mode groups values into half-hour buckets, maintains rolling sums over the selected horizon, and evaluates the final boolean once enough data has accumulated. Debounce checks prevent repeated alerts for the same tracking key.

Analysis batches publish notification evaluation only after the full batch has left queued or processing states. A transaction-backed lock on the batch document prevents duplicate notification evaluation messages, even when multiple workers finish near the same time. Delivery currently includes idempotent in-app notifications, with email and SMS left as planned channel implementations.

Security

Rules protect user data while server functions enforce product limits.

Firestore security rules are used as the client-facing access boundary. Admins can access administrative data, users can read and write their own workflow, post, prompt, aggregator, notification, and statistics documents, and authenticated users can read shared post data and deltas. Sensitive fields such as credits, email, Discord identifiers, subscription state, and protected profile fields are readable where appropriate but server-managed for writes.

Some settings are intentionally saved through Cloud Functions instead of direct client writes. Workflow saving, run control, aggregator creation, billing reconciliation, and account linking all happen server-side so the backend can validate roles, reject admin-only advanced settings, enforce subscription limits, maintain counters transactionally, and keep protected Firestore fields out of the client's write surface.

Supporting Systems

Billing, aggregators, cleanup, and observability complete the loop.

Stripe reconciliation keeps subscription state mirrored onto the user document, while analysis billing tracks credit spend at the prompt level so retries do not double-charge completed prompts. Aggregator creation and deletion use transactions to enforce plan limits and keep per-user counters accurate.

Cleanup is handled with deterministic Cloud Task IDs so workflow stop and expiration can be rescheduled safely. Scrape runs also write heartbeat documents with progress, pending delta count, heavy write count, analysis queue count, and bulk write error counts so the dashboard can show activity without creating a hot document on every post.

Cloud Functions

Node.js functions coordinate workflow configuration, analysis, billing, notifications, and admin operations. Python functions handle Reddit scraping and Discord OAuth.

Pub/Sub + Cloud Tasks

Pub/Sub starts broad asynchronous work; Cloud Tasks provides controlled HTTP workers, deterministic task IDs, retries, queue pausing, and delayed cleanup.

Firestore

Used as product database, shared cache, scheduler registry, bot lease table, queue store, billing ledger, and per-user access boundary.

Secret Manager

Stores Discord, Reddit, and LLM credentials outside the codebase and loads them only from server-side functions.

Technology Summary

What this project demonstrates.

Daemon Data demonstrates multi-provider authentication, serverless orchestration, multi-tenant Firestore modeling, transactional resource leasing, rate-limit-aware scraping, queue-based LLM processing, credit billing, Firestore security rules, and graph-based automation design.
